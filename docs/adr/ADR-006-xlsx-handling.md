# ADR-006 — DuckDB SQL path alongside vector path for XLSX

- **Status:** Accepted
- **Date:** 2026-04-23

## Context

Spreadsheets are **fundamentally different** from prose documents. Questions users ask include:

- "Which project in 2024 had the highest margin?"
- "Sum the costs where supplier = X."
- "Average delivery time for all orders in Q2."

These are **numeric / aggregate** questions. Vector retrieval returns *text chunks that look relevant* — useful to locate the right sheet, useless to compute an answer. An LLM asked to sum cells in a prompt often gets numbers subtly wrong, especially in German locale (`,` vs `.` decimals).

Treating XLSX purely as text is the core reason the n8n starter kit fails on spreadsheet questions.

## Decision

At ingestion time, every XLSX sheet (and every confidently-extracted PDF table) is loaded into **DuckDB** as a typed columnar table, in addition to being chunked + embedded like any other document. The DuckDB database is a single file at `${DATA_ROOT}/duckdb/reineke.duckdb`.

At query time, a **classifier** step decides whether the query is "table-math" (words like *größte, sum, mittel, average, how many, welche …*, explicit file mention with `.xlsx`, numeric comparators). If yes, we run a parallel SQL branch:

1. LLM (reasoning tier) is given: the tables' schemas (only those the user is allowed to see) + the question, and asked to produce a single `SELECT`.
2. `duckdb-api` parses and validates the statement (SELECT only, no ATTACH, no COPY, no UDFs outside a tiny whitelist, no filesystem access).
3. It executes against an ACL-filtered view (`views.v_<table>_<group_hash>`) — writes baked into the SQL path, cannot be bypassed.
4. Rows + column list are returned.

The text path (vector + rerank) runs in parallel. Final prompt to the answer-LLM gets **both** — SQL result as a table, text chunks as qualitative context — and is asked to answer citing both when relevant.

## Why DuckDB specifically

- Embedded, file-based, zero-ops.
- Excellent at small-to-medium analytical workloads.
- Mature SQL surface; users with a bit of SQL know it.
- Columnar: sampling schema is cheap.
- Python bindings that we use in the worker; HTTP in the service.

## Alternatives considered

- **Postgres tables**: would share the metadata instance; but co-locating user-data analytics with auth/audit raises operational coupling, and Postgres is row-oriented (slower for aggregates at scale).
- **Pandas on demand**: requires loading the whole sheet into the worker per query; DuckDB is faster and already typed.
- **"Just pass the raw CSV to the LLM"**: classic failure mode; numbers are read out loud with errors. Rejected.
- **Text-to-SQL against a proper OLAP (ClickHouse, DuckDB-on-disk in shared server mode)**: overkill for this scale.

## Consequences

Positive:

- Numerical answers are *computed*, not generated; citations include the executed SQL so users can verify.
- Maintenance is trivial — DuckDB is a file; backup is `cp`.

Negative:

- Text-to-SQL correctness depends on the schema the LLM sees. We show **typed column names + one sample value per column** — a known-good pattern that keeps prompts short.
- DuckDB is single-writer; only ingestion-worker writes, which is fine for our architecture. Multiple readers (retrieval via the duckdb-api) are fine.
- Column inference from messy Excel is imperfect. We keep the raw XLSX too; if the column types are wrong, the operator fixes the sheet or adds a small cleanup step in the ingestion pipeline.

## Safety

The SQL executor is **not** a general shell:

- Single-statement enforcement.
- Only `SELECT` allowed.
- No `ATTACH`, `COPY`, `INSTALL`, `LOAD`, `PRAGMA set_variable`.
- UDF whitelist: none by default; numeric aggregates only.
- Statements are **generated by an LLM**; they are *not* trusted blindly. The executor is the trust boundary; the ACL view is in its SQL, not optional.

If the classifier + planner fail (LLM produces invalid SQL), the text-only path still answers; the UI shows a small note "SQL branch did not produce a confident plan — answered from text only."
